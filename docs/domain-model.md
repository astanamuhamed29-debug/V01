# Domain Model — SELF-OS

This document defines the canonical domain objects used throughout SELF-OS. These are
conceptual definitions — the authoritative specification of what each entity means
in the domain. Concrete implementations may be found in the modules referenced for each
object.

---

## Canonical Domain Objects

### MemoryEvent

A single unit of information that has been stored in the system's memory. A MemoryEvent
may originate from a user message, an agent observation, a tool result, an external
import, or an inferred insight.

**Fields**: `id`, `user_id`, `timestamp`, `content`, `node_type`, `tags`, `source`,
`embedding`, `abstraction_level`, `confidence`, `relationship_refs`

**Graph representation**: A typed node in the knowledge graph. Node type may be EVENT,
INSIGHT, MEMORY, CONCEPT, or others. Relationships connect it to related nodes (people,
topics, emotions, goals).

**Implementation**: `core/graph/storage.py`, `core/graph/api.py`

---

### EmotionState

A snapshot of the user's emotional state at a specific moment, represented using the
VAD (Valence-Arousal-Dominance) model along with discrete emotion labels.

**Fields**: `user_id`, `timestamp`, `valence` (-1..+1), `arousal` (-1..+1),
`dominance` (-1..+1), `emotion_labels` (list of strings), `intensity`, `confidence`

**Graph representation**: Stored as mood snapshots in the graph. Also embedded in
PsycheState.

**Implementation**: `core/mood/tracker.py`

---

### Need

A fundamental psychological need that the user has expressed or that has been inferred
from their behaviour and language patterns. Based on frameworks such as NVC
(Non-Violent Communication) and self-determination theory.

**Fields**: `id`, `user_id`, `label` (e.g. "autonomy", "connection", "recognition"),
`intensity`, `unmet`, `evidence_refs`, `created_at`, `updated_at`

**Graph representation**: NEED node in the knowledge graph. Connected to related
messages, events, and parts.

**Implementation**: `core/graph/` (NEED node type), `core/parts/memory.py`

---

### Belief

A persistent belief or mental model the user holds about themselves, others, or the
world. May be functional or dysfunctional. Inferred from repeated patterns in language
and behaviour.

**Fields**: `id`, `user_id`, `content` (natural language statement), `category`
(e.g. "self-concept", "world-model", "relational"), `valence`, `confidence`,
`evidence_refs`, `created_at`, `updated_at`

**Graph representation**: BELIEF node in the knowledge graph.

**Implementation**: `core/graph/` (BELIEF node type)

---

### Value

A core personal value that the user holds and that guides their decisions and goals.
Values are extracted through onboarding and inferred from behavioural patterns.

**Fields**: `id`, `user_id`, `label` (e.g. "creativity", "integrity", "family"),
`importance` (0..1), `evidence_refs`, `created_at`

**Graph representation**: VALUE node in the knowledge graph.

**Implementation**: `core/graph/` (VALUE node type)

---

### Part

An IFS (Internal Family Systems) sub-personality or internal part that influences the
user's behaviour, emotional responses, and decision-making. Parts include protectors
(managers and firefighters) and exiles.

**Fields**: `id`, `user_id`, `key`, `subtype` (manager/firefighter/exile/self),
`voice_prompt`, `activation_level`, `last_activated_at`, `triggers`, `evidence_refs`

**Graph representation**: PART node in the knowledge graph. Also tracked in
PartsMemory for real-time activation.

**Implementation**: `core/parts/memory.py`, `agents/ifs/parts.py`

---

### Goal

A persistent, user-defined (or agent-inferred) objective that the user is working toward.
Goals may be hierarchically decomposed into sub-goals and tasks. Goals have a status
lifecycle: active → paused → completed / abandoned.

**Fields**: `id`, `user_id`, `title`, `description`, `status`, `priority`, `deadline`,
`parent_goal_id`, `sub_goals`, `tasks`, `evidence_refs`, `created_at`, `updated_at`

**Graph representation**: GOAL node in the knowledge graph. TASK nodes are linked to
goals.

**Implementation**: `core/goals/engine.py`

---

### Task

A concrete, actionable item derived from a goal or identified proactively by the agent.
Tasks are smaller in scope than goals and are expected to be completable in a single
work session.

**Fields**: `id`, `user_id`, `title`, `description`, `status`, `goal_id`,
`due_date`, `priority`, `created_at`, `updated_at`

**Graph representation**: TASK node in the knowledge graph.

**Implementation**: `core/tools/task_tool.py`, `core/goals/engine.py`

---

### Insight

A higher-order pattern, observation, or inference generated by the system from
accumulated memory. Insights are produced by the analytics engine, the InnerCouncil,
or the reflection pipeline.

**Fields**: `id`, `user_id`, `content`, `insight_type` (e.g. "pattern", "prediction",
"recommendation"), `evidence_refs`, `confidence`, `created_at`

**Graph representation**: INSIGHT node in the knowledge graph.

**Implementation**: `core/insights/`, `core/analytics/`

---

### PsycheState

A complete, point-in-time snapshot of the user's psychological and cognitive state.
Integrates emotional, parts, cognitive, and goal dimensions into a single serialisable
object. This is the primary state contract used by all agent logic.

**Fields**: `user_id`, `timestamp`, `valence`, `arousal`, `dominance`,
`dominant_label`, `active_parts`, `dominant_part`, `dominant_need`, `active_beliefs`,
`active_goals`, `cognitive_load`, `cognitive_distortions`, `stressor_tags`,
`brain_activation`, `dominant_neuron_type`, `confidence`

**Note**: There are two PsycheState implementations in the codebase:
- `core/psyche/state.py` — user-facing, 15+ fields, used by the therapy and tool layers
- `core/prediction/state_model.py` — prediction-layer DTO, 9 core fields, with
  `from_brain_state` / `to_brain_state` conversion methods

Both are dataclasses. The prediction-layer DTO is a leaner projection used by the
PredictiveEngine; the user-facing version is the authoritative state object.

**Implementation**: `core/psyche/state.py`, `core/prediction/state_model.py`

---

### MotivationState

A structured representation of the user's current motivational context: what they are
motivated to do, what is blocking them, what needs are unresolved, and what the agent
should do proactively. Derived from PsycheState, active goals, emotional state, and
value tensions.

**Fields**: `user_id`, `timestamp`, `active_goals`, `unresolved_needs`,
`dominant_emotions`, `value_tensions`, `priority_signals`, `action_readiness`,
`recommended_next_actions`, `constraints`, `evidence_refs`, `confidence`

**Implementation**: `core/motivation/schema.py`, `core/motivation/builder.py`

---

### AgentAction

A record of a specific action taken by the agent on behalf of the user. Actions are
first-class objects that are persisted with their triggering context, motivation
references, memory references, tool calls, result, and explanation. This enables
auditability and learning from past actions.

**Fields**: `id`, `user_id`, `timestamp`, `action_type`, `title`, `description`,
`status`, `triggered_by`, `motivation_refs`, `memory_refs`, `tool_calls`, `result`,
`explanation`

**Implementation**: `core/agent/schema.py`

---

### IdentityProfile

A structured summary of the user's identity, synthesised from their values, beliefs,
needs, parts, goals, and behavioural patterns. The IdentityProfile is the output of the
onboarding process and is continuously updated as new information is acquired.

**Fields**: `user_id`, `display_name`, `values` (list of Value), `beliefs` (list of
Belief), `needs` (list of Need), `parts` (list of Part), `active_goals` (list of Goal),
`domain_profiles` (list of DomainProfile), `completeness_score`, `created_at`,
`updated_at`

**Implementation**: `core/identity/schema.py` (`IdentityProfile`). Built by
`core/identity/builder.py` (`IdentityProfileBuilder`) from live graph nodes.

---

### DomainProfile

A domain-specific identity sub-profile capturing the user's knowledge, experience,
preferences, and goals within a particular domain (e.g. "software engineering",
"creative writing", "health and fitness"). Domain profiles are populated during
onboarding and refined continuously.

**Fields**: `domain`, `user_id`, `knowledge_level`, `experience_years`, `preferences`,
`goals`, `notes`, `confidence`, `created_at`, `updated_at`

**Implementation**: `core/identity/schema.py` (`DomainProfile`). Populated by
`IdentityProfileBuilder` from PROJECT, TASK, BELIEF, NEED, INSIGHT, and VALUE graph nodes.

---

### ProfileGap

A detected incompleteness in the user's identity or domain profile. Profile gaps are
identified by comparing the current IdentityProfile against expected completeness
criteria for each domain. Gaps drive adaptive interviewing in onboarding.

**Fields**: `user_id`, `domain`, `gap_type` (e.g. "missing_value", "missing_goal",
"low_confidence_belief"), `description`, `priority`, `detected_at`

**Implementation**: `core/identity/schema.py` (`ProfileGap`). Generated automatically by
`IdentityProfileBuilder` during `build()`. See [docs/onboarding-and-identity-bootstrap.md](onboarding-and-identity-bootstrap.md).

A structured session in which the system acquires initial identity information from a
new user. An onboarding session consists of multiple question/answer exchanges, each
contributing to the population of the IdentityProfile and DomainProfile.

**Fields**: `id`, `user_id`, `domain`, `status` (active/complete/paused),
`questions_asked`, `answers_received`, `gaps_addressed`, `confidence_delta`,
`started_at`, `completed_at`

**Implementation**: Planned. See [docs/onboarding-and-identity-bootstrap.md](onboarding-and-identity-bootstrap.md).

## Relationship to Graph-Based Storage

All canonical domain objects above are either:

1. **Directly stored as graph nodes** (MemoryEvent, Need, Belief, Value, Part, Goal,
   Task, Insight) — each with a `node_type` that matches its domain concept.

2. **Derived views over graph nodes** (EmotionState → mood snapshots, PsycheState →
   aggregated from multiple node types, MotivationState → synthesised from PsycheState
   and graph queries).

3. **Action records** (AgentAction) — persisted separately as operation logs.

4. **Profile aggregations** (IdentityProfile, DomainProfile, ProfileGap,
   OnboardingSession) — planned as higher-level aggregations over existing graph data.

The graph is the **source of truth** for all persistent knowledge. All higher-level
objects are computed from it and should not be stored as separate authoritative sources.

### Node Type Mapping

| Domain Object | Graph Node Type |
|---|---|
| MemoryEvent | EVENT, MEMORY, CONCEPT, OBSERVATION |
| Need | NEED |
| Belief | BELIEF |
| Value | VALUE |
| Part | PART |
| Goal | GOAL |
| Task | TASK |
| Insight | INSIGHT |
| EmotionState | (mood snapshot table, not a graph node) |
| PsycheState | (derived view, not stored as a node) |
| MotivationState | (derived view, not stored as a node) |
| AgentAction | (action log, not a graph node) |
